프로젝트 전체 코드를 점검하고, 반복/중복 코드를 찾아 우선순위별로 리팩토링해줘.

## 진행 순서

---

### 1단계 — 코드 전체 탐색 (Survey)

다음 항목을 모두 확인한다:

- `src/` 하위의 **모든 페이지, 컴포넌트, 훅, 유틸, API 라우트** 파일 목록과 대략적인 크기 파악
- **중복 로직** 탐색:
  - 동일하거나 거의 유사한 함수/컴포넌트가 2곳 이상에 정의된 것
  - 비슷한 useState + useEffect 패턴이 여러 페이지에 흩어진 것
  - inline으로 반복 작성된 유틸 함수 (날짜 계산, API 호출 boilerplate 등)
- **타입 중복** 탐색:
  - 같은 형태의 인터페이스/타입이 여러 파일에 각각 선언된 것
- **UI 패턴 중복** 탐색:
  - 동일한 컴포넌트 UI가 2개 이상 파일에 인라인 함수로 정의된 것

---

### 2단계 — 중복/개선 항목 정리 및 우선순위 분류

발견한 항목을 아래 기준으로 분류해 목록으로 제시한다:

**🔴 높은 우선순위** (즉시 리팩토링):
- 3곳 이상에서 반복되는 핵심 로직
- 타입 정의 중복 (공유 타입 파일로 통합 가능한 것)
- 비즈니스 로직 관련 유틸 함수 중복

**🟡 중간 우선순위** (다음 단계 리팩토링):
- 2곳 이상 페이지의 공통 상태/이벤트 로직 → 커스텀 훅으로 추출 가능한 것
- 반복 UI 블록 → 공통 컴포넌트로 추출 가능한 것

**🟢 낮은 우선순위** (여유 있을 때):
- 2곳 정도의 소규모 중복 (유틸 함수, 서브 컴포넌트 등)
- 가독성 개선 목적의 정리

각 항목에 대해 **파일명, 중복 내용, 제안하는 해결 방법**을 명시한다.

---

### 3단계 — 리팩토링 실행

우선순위 **높은 것 → 중간 → 낮은 것** 순서로 실행한다.

각 우선순위 그룹을 실행하기 전에 사용자에게 해당 그룹의 작업 목록을 먼저 보여주고 진행 여부를 확인한다.

실행 시 지켜야 할 규칙:
- 기존 동작을 바꾸지 않는다 — 리팩토링은 구조 개선이지 기능 변경이 아니다
- 추출한 파일의 위치:
  - 공용 유틸 → `src/lib/utils/`
  - 공용 훅 → `src/lib/hooks/`
  - 공용 컴포넌트 → `src/components/common/` 또는 관련 도메인 폴더
  - 공용 타입 → `src/types/`
- 추출 후에는 기존 파일에서 로컬 정의를 제거하고 import로 교체한다
- 각 그룹 완료 후 반드시 `npx tsc --noEmit`으로 타입 오류가 없는지 확인한다

---

### 4단계 — 완료 보고

리팩토링이 끝나면 아래 형식으로 요약한다:

```
## 리팩토링 완료 요약

### 새로 생성한 파일
- `경로` — 설명

### 수정한 파일
- `경로` — 변경 내용

### 제거된 중복
- 내용 (몇 곳 → 1곳으로 통합)
```
